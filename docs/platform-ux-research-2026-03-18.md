# 运维支撑平台探索笔记

日期：2026-03-18

目标：基于真实平台探索结果，为 `easyconnect-workbench` 后续重构提供产品和交互依据，而不是凭印象设计。

## 优先级修正

用户后来明确补充：

- `提测` 不是当前聚合应用的核心
- 核心痛点是 `测试环境 / 正式环境` 下的构建任务流转

真实主流程应调整为：

1. 在 `我的应用 -> 服务列表 -> 服务详情` 中为测试或正式环境创建构建任务
2. 再切到 `构建服务管理` 找到对应申请/任务
3. 在该页继续构建
4. 构建成功后发布
5. 构建失败后查看原因，再继续构建

也就是说，当前平台最反人类的地方不是“提测表单本身”，而是：

- 创建任务和执行构建分散在不同页面
- 构建和发布不在同一个稳定工作面板里
- 失败排查后再继续构建仍然要重复找上下文
- 用户每次都要从 `我的应用` 重新疯狂点击很多层

后续聚合应用的首要目标应该是把这条链路做成一个连续工作流。

## 探索范围

- 登录进入真实构建平台
- 从 `我的应用 -> 服务列表 -> 服务详情 -> 构建`
- 从 `测试环境 -> 提测构建 -> 提测表单 -> 提测记录表 -> release构建任务`
- 从 `构建中心` 顶部菜单抽样进入：
  - `我的微服务`
  - `构建服务管理`
  - `查看服务清单`

## 已确认的真实信息架构

### 顶部导航

`构建中心` 本身是一个下拉入口，不是单一页面。菜单包括：

- 服务管理
- 我的应用
- 我的微服务
- 申请微服务
- 组件接入情况
- 审批管理
- 构建服务管理
- 新建服务管理

### 我的应用链路

1. `我的应用`
2. 按医院 / 应用进入 `服务列表`
3. 在 `服务列表` 内继续按环境切换：
   - 开发环境
   - 测试环境
   - 生产环境
   - 生产测试环境
4. 每个服务再进入：
   - `构建`
   - `查看服务详情`
5. 在测试环境中还会分叉出：
   - `提测构建`
   - `提测记录表`
6. `提测记录表` 继续进入：
   - `release构建任务`
7. `release构建任务` 继续进入单服务构建日志和阶段详情

### 另一条链路

`构建中心 -> 构建服务管理` 是另一套列表模型，偏待发布 / 处理单视角，不是应用视角。

这一页存在：

- 待处理 / 已退回 / 已处理
- 现场发布
- 查看服务清单

`查看服务清单` 继续进入一层服务级任务页，右侧仍然是阶段时间和控制台日志。

## 实际操作结论

### 1. 服务构建不是“任务中心”模型，而是混合页

在 `服务详情` 页面点击 `构建` 后：

- 页面右上角动作会从 `构建` 变成 `停止构建`
- 中间区域同时开始加载历史阶段耗时
- 右侧同时显示控制台日志
- 页面不会跳到一个清晰的“当前任务页”

这意味着：

- 历史详情
- 当前任务状态
- 操作入口

被混在同一页里，语义不清。

### 2. 返回路径会丢失上下文

这是本次最关键的 UX 发现之一。

已实测两次：

- 从 `测试环境 -> 某 release 服务详情` 返回后，系统没有回到原测试环境列表，而是回到了开发环境默认列表
- 从 `测试环境 -> 提测构建表单` 返回后，系统再次回到了开发环境默认列表

这说明当前平台没有真正维护“用户所在上下文”，而是用默认页兜底。

对用户的真实体验影响：

- 重新找医院
- 重新找应用
- 重新切环境
- 重新定位服务

这正是后续聚合应用必须消灭的行为。

### 3. `提测` 链路存在，但不是当前 V1 的核心

`测试环境` 页面点击 `提测构建` 后，并不会直接进入任务单，而是先出现一层类型说明：

- 提测构建
- 修复构建
- hotfix 合并

每种类型都再有一个 `构建` 按钮。

继续点击后，才进入真正的提测表单页。

### 4. 提测表单是另一套页面模型

提测表单要求填写：

- 提测子应用
- 提测模块
- 测试类型
- 提测范围/重点

同时还要在同一页完成：

- 从微服务清单中勾选服务
- 添加到已选服务
- 检查构建版本

这说明当前平台把“任务元信息”和“服务选择器”挤在一页里，信息负担很高。

### 5. 提测记录表和 release 构建任务又是两层

在 `提测记录表` 中会出现记录卡片，例如：

- 住院4月第二版
- 住院
- 住院3月第一次迭代

点击 `查看详情` 后进入 `release构建任务` 页面。

在这页里：

- 左侧是任务列表
- 中部上方是阶段耗时
- 右侧是控制台日志

结构上和服务详情页类似，又是一种“列表 + 阶段 + 日志”的混合页。

### 6. `构建服务管理` 才是构建任务真正继续流转的关键页面

用户补充后，`构建服务管理` 的重要性被确认高于 `提测`。

这个页面不再围绕应用和环境，而是围绕申请单/处理单：

- 申请列表
- 处理状态
- 现场发布
- 查看服务清单

这和 `我的应用 -> 服务列表` 的应用视角完全不同，但现实操作中用户必须在两者之间来回切。

用户在不同页面之间需要不停切换心智模型：

- 现在我是按应用看？
- 按服务看？
- 按提测记录看？
- 按发布申请单看？

## 对聚合应用的直接启发

V1 不应该复刻网页结构，而应该重新抽象为稳定工作流：

1. 连接层
   - VPN 状态
   - 平台登录状态
2. 上下文层
   - 当前医院
   - 当前业务应用
   - 当前环境
3. 任务层
   - 在当前上下文里创建测试/正式构建任务
   - 查看任务当前状态
   - 跳转到待处理/待发布任务
4. 执行层
   - 继续构建
   - 查看失败原因
   - 构建成功后发布
5. 结果层
   - 最近任务
   - 正在构建
   - 待发布
   - 失败待处理
   - 发布结果

最重要的是：

- 保留上下文，不允许返回后丢环境
- 把“创建构建任务 -> 执行构建 -> 发布/重试”做成同一条连续链路
- 把日志、阶段、失败原因、重试入口放进同一个任务详情，而不是散在多层页面
- 提供最近项目和最近环境的快速回到能力
- 提供“我刚创建的任务”直接可见，不需要再去另一页查找
