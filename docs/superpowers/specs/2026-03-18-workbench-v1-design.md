# EasyConnect Workbench V1 Design

日期：2026-03-18

状态：已与用户对齐，待进入实现计划

## 1. 背景

当前真实工作流被拆散在多个系统和多个页面层级里：

- VPN 需要依赖本地 EasyConnect 客户端
- 构建平台负责服务级构建、构建任务、公司发布入口
- 发布平台负责按医院、环境、业务组做现场发布

用户的核心痛点不是“不会操作”，而是：

- 每次都要从 `我的应用` 开始不断点层级
- 同一条业务链路被拆到不同页面、不同心智模型中
- 返回时会丢失上下文，必须重新定位医院、应用、环境和服务
- 常用服务的补丁构建、补丁发布、失败重试成本过高

目标不是复刻网页，而是做一个以“服务构建并发布”为中心的桌面工作台。

## 2. 设计目标

### 2.1 主目标

- 打开应用后直接回到上次工作上下文
- 让用户围绕“服务”而不是“平台页面”工作
- 在一个工作台内完成构建、重试、发布、克隆历史任务
- 把 VPN、构建平台、发布平台的登录态收敛成后台连接层

### 2.2 成功标准

- 用户不需要手动来回切构建平台和发布平台完成主链路
- 开发、测试、生产三种环境的差异被系统吸收，不再靠用户记忆
- 常用服务能直接从列表一键继续工作
- 历史任务能快速克隆成新任务，用于补丁测试和补丁发布

### 2.3 非目标

- V1 不追求覆盖原平台所有后台管理功能
- V1 不优先做复杂批量编排
- V1 不优先做人工星标体系，先按最近使用自动排序
- V1 不把连接层做成独立业务模块

## 3. 关键事实

### 3.1 真实数据关系

顶部上下文必须是级联关系：

`医院 -> 业务应用 -> 环境 -> 服务`

原因：

- 每个医院可用的业务应用并不完全一致
- 环境和服务都依赖当前医院与业务应用

发布平台业务组映射：

- Workbench 不单独把“业务组”暴露成主上下文字段
- 发布平台中的业务组默认从当前 `业务应用` 推导
- 默认选择策略：
  - 优先精确匹配当前业务应用代码或名称
  - 其次做规范化别名匹配（大小写、分隔符归一）
  - 若仍无法唯一确定，则不自动现场发布，直接进入 `阻塞` 并引导手动接管

V1 边界：

- 业务组别名匹配只使用内置规范化规则
- V1 不提供用户自定义业务组映射配置
- 若规范化后仍无法唯一确定，则直接进入 `阻塞`
- 业务组自定义映射属于后续版本能力

统一标识规则：

- Workbench 内部不直接依赖某个平台的单一 ID，而是维护一套规范化主键：
  - `hospitalKey`
  - `appKey`
  - `serviceKey`
  - `envKey`

主键生成优先级：

- 优先使用平台稳定代码或显式 ID
- 若无稳定代码，则使用规范化名称
- 若不同平台无法唯一归一，则进入 `阻塞`，不允许自动链路继续

规范化名称规则：

- 去前后空格
- 转小写
- 合并连续空白
- 去除常见分隔符影响，例如 `-`、`_`

V1 不允许“模糊猜测成功就继续执行”。
任何跨平台映射若无法唯一确定，都必须转为阻塞或手动接管。

### 3.2 三种环境的真实主链路

开发环境：

- 直接构建
- 构建成功后自动发布到开发环境

测试环境：

- 创建构建任务
- 执行构建
- 现场发布

生产环境：

- 创建构建任务
- 执行构建
- 公司发布
- 切换到发布平台
- 现场发布

### 3.3 用户工作模式

- 主要处理 1 到 2 个服务
- 常见场景是对常用服务反复构建、重试、发布
- 历史任务经常需要再次复用，作为补丁测试或补丁发布的模板

### 3.4 任务建模约束

- V1 默认以 `单服务任务` 为主
- 只有用户主动选择 `新建任务` 或 `克隆历史任务` 时，才允许创建多服务任务
- 同一 `服务 + 环境` 在任一时刻只存在一个当前有效状态
- 历史失败任务、历史成功任务都归档到历史任务，不参与当前状态竞争

## 4. 产品信息架构

### 4.1 一级导航

V1 只保留两个左侧一级菜单：

- 工作台
- 设置

设计原则：

- 默认所有高频能力都留在工作台内
- 只有真正独立且长期存在的功能才升级为一级菜单

### 4.2 工作台结构

工作台采用稳定的三段式布局：

- 顶部：上下文条
- 中间：服务工作区
- 右侧：轻量任务侧栏

用户绝大多数操作都应在该页面内完成，不再依赖跨页面跳转。

## 5. 工作台详细设计

### 5.1 顶部上下文条

包含以下元素：

- 医院选择器
- 业务应用选择器
- 环境切换器
- 右侧连接状态区

行为规则：

- 启动应用时恢复上次的 `医院 + 业务应用 + 环境`
- 切换医院时，业务应用列表联动刷新
- 如果上次记录的业务应用仍然存在，则自动恢复
- 不存在则选择该医院最近使用的业务应用
- 切换环境时，不丢当前医院和业务应用
- 每个 `医院 + 业务应用 + 环境` 组合单独记住：
  - 搜索词
  - 列表滚动位置
  - 最近操作服务

最近使用定义：

- 触发以下行为之一，即记为一次“使用”：
  - 点击服务行主动作
  - 从服务行打开任务详情
  - 对该服务执行 `新建任务`
  - 对该服务执行 `克隆历史任务`
- 最近使用按 `lastActionAt` 倒序排序

### 5.2 中间服务工作区

主工作区永远显示当前上下文下的服务列表。

V1 只保留两个核心筛选：

- 服务名搜索
- 状态筛选：全部 / 进行中 / 失败 / 待发布

筛选口径：

- `全部`：包含所有当前服务投影状态
- `进行中`：包含 `进行中` 以及等待平台结果的运行态
- `失败`：包含 `失败待处理` 与 `阻塞`
- `待发布`：仅包含 `待发布`
- `待执行` 与 `已完成` 不单独提供筛选项，归入 `全部`

排序优先级：

1. 最近使用
2. 进行中
3. 失败待处理
4. 待发布
5. 其他

工作区级任务入口：

- 在服务列表工具栏提供一个显式 `新建任务` 入口
- 该入口始终作用于当前 `医院 + 业务应用 + 环境`
- 用于创建多服务任务，或显式创建“仅创建不立即执行”的任务

### 5.3 右侧任务侧栏

侧栏常驻但可收起，分成四组：

- 进行中
- 失败待处理
- 待发布
- 最近任务

作用：

- 提醒，不抢主视图
- 让用户快速回到当前关键任务
- 给历史任务提供克隆入口

展示粒度规则：

- 侧栏以“任务”为单位展示，而不是以“服务”为单位展示
- 多服务任务在侧栏只出现一次
- 每个任务条目至少展示：
  - 任务 ID 或任务名称
  - 服务数
  - 当前最关键摘要，例如首个失败服务或当前执行阶段

分组规则：

- 同一任务同一时刻只能出现在一个分组中
- 分组优先级为：
  - 失败待处理/阻塞
  - 进行中（含排队/等待）
  - 待发布
  - 最近任务

保留策略：

侧栏与历史任务统一遵守以下保留规则：

- `最近任务` 仅展示当前 `医院 + 业务应用 + 环境` 最近 20 条任务
- 任务详情抽屉中的历史任务默认展示当前 `服务 + 环境` 最近 30 条
- 本地历史缓存最多保留 `180 天或 2000 条`
- 任何仍作为当前状态来源的任务不得被清理

## 6. 服务行设计

每一行服务不是纯展示行，而是一个轻量工作单元。

建议包含以下信息：

- 服务名称
- 服务中文名
- 当前环境版本
- 最近构建时间
- 最近发布时间
- 当前任务状态
- 失败摘要
- 主动作
- 次动作

最小数据来源要求：

- 当前环境版本：来自当前上下文的服务状态数据
- 最近构建时间：来自构建平台任务或服务状态数据
- 最近发布时间：
  - `dev`：来自开发环境最近一次自动发布结果
  - `test/prod`：优先来自发布平台现场发布时间
- 当前任务状态：来自服务投影
- 失败摘要：来自当前任务最近失败阶段

### 6.1 主动作规则

开发环境：

- `构建`

说明：

- `dev` 环境不存在 `待发布`
- `dev` 的成功链路终点是 `已完成`

测试环境：

- 无任务时显示 `开始构建`
- 或 `继续构建`
- 或 `现场发布`

生产环境：

- 无任务时显示 `开始构建`
- 或 `继续构建`
- 或 `生产发布`

这里的 `生产发布` 对用户是单次点击，对系统是整条自动链路：

`公司发布 -> 发布平台现场发布`

“开始构建”的语义：

- 对开发环境：直接开始构建
- 对测试/生产环境：若当前无任务，则隐式先创建任务，再自动进入构建阶段
- 也就是说，服务行主路径永远优先“少一次点击”，不要求用户先单独完成任务创建

`待执行` 状态下的主动作：

- 若服务投影状态为 `待执行`，主动作统一显示为 `开始执行`
- 点击后从该任务的第一个未执行阶段继续
- 不会重新创建任务

按环境的主按钮推进边界：

- `dev`
  - 主按钮：`构建`
  - 点击后自动推进：`构建 -> 自动发布到开发环境`
  - 成功后直接进入 `已完成`
- `test`
  - 无任务：`开始构建`
  - 构建失败：`继续构建`
  - 待发布：`现场发布`
  - 点击 `开始构建/继续构建` 只自动推进到“构建完成”
  - 构建完成后停在 `待发布`
  - 点击 `现场发布` 才进入发布阶段
- `prod`
  - 无任务：`开始构建`
  - 构建失败：`继续构建`
  - 待发布且命中 allowlist：`生产发布`
  - 待发布且未命中 allowlist：`公司发布`
  - 点击 `开始构建/继续构建` 只自动推进到“构建完成”
  - `生产发布` 才会执行 `公司发布 -> 现场发布`
  - `公司发布` 只执行到公司发布完成，随后进入 `阻塞`

### 6.2 次动作

次动作不占用主按钮位，收敛为：

- 日志
- 任务详情
- 重试当前步骤
- 重试整条链路
- 新建任务
- 克隆历史任务

`新建任务` 与服务行关系：

- 从服务行进入 `新建任务` 时，默认预选当前服务
- 用户可以在任务创建器中添加或移除其他服务
- 该入口用于“当前任务不合适，需要另开一单”的场景

说明：

- 若当前失败阶段正好是构建阶段，界面允许把 `重试当前步骤` 文案别名展示为 `重试构建`
- 但命令语义仍然统一为“重试当前步骤”

## 7. 任务模型

### 7.1 任务定义

任务是一等对象，不再依赖原平台的页面层级来理解。

一个任务至少包含：

- 医院
- 业务应用
- 环境
- 服务范围
- 当前状态
- 当前阶段
- 最近一次执行日志
- 最近一次失败原因
- 集成标识符：
  - 构建平台任务标识
  - 发布平台任务标识（如存在）
  - 对账与手动接管所需的外部 ID

V1 任务分两类：

- 默认任务：单服务任务
- 显式任务：用户主动创建或克隆得到的多服务任务

术语定义：

- 服务：当前 `医院 + 业务应用 + 环境` 下的单个可构建单元
- 任务：一次可执行的构建/发布工作对象
- 阶段：任务执行链中的单个步骤，例如 `构建`、`公司发布`、`现场发布`
- 当前状态：当前 `服务 + 环境` 的唯一有效状态，用于服务行展示
- 历史任务：已经完成、失败、作废或被替换的旧任务，仅用于查看和克隆
- 服务投影：把任务状态映射到某个 `医院 + 业务应用 + 环境 + 服务` 上的一条当前有效展示记录

### 7.2 环境阶段模型

开发环境：

- 构建
- 自动发布到开发环境

测试环境：

- 创建任务
- 构建
- 现场发布

生产环境：

- 创建任务
- 构建
- 公司发布
- 现场发布

说明：

- 生产环境允许默认全自动推进整条链路
- 不额外等待人工确认
- `生产发布` 只是前端主按钮名，后台实际串行执行 `公司发布 -> 发布平台现场发布`

生产自动化安全开关：

- V1 必须提供生产自动发布总开关
- V1 必须支持按 `医院 + 业务应用 + 环境` 维度配置生产自动发布 allowlist
- 默认值：
  - 总开关默认关闭
  - allowlist 默认为空
- 在 allowlist 内的上下文，生产链路可以全自动执行且不再逐次确认
- 不在 allowlist 内时：
  - 不执行自动现场发布
  - 但仍允许执行 `公司发布`
  - `公司发布` 成功后，任务进入 `阻塞`
  - 阻塞原因明确标记为“当前上下文未启用生产自动现场发布”
  - 后续只引导手动接管，不在 V1 内继续自动现场发布

### 7.3 失败与阻塞

失败和阻塞要明确区分。

失败：

- 某一阶段执行出错
- 允许 `重试当前步骤` 或 `重试整条链路`

阻塞：

- 例如当前应用仍有旧版本未发布
- 当前任务状态不允许继续推进
- 需要先处理前置条件

阻塞必须直接用人话显示原因与下一步建议，不能只展示红色状态。

### 7.4 状态来源与优先级

服务行状态、任务状态、阶段状态分三层：

- 服务行状态：当前 `服务 + 环境` 的唯一有效状态
- 任务状态：当前任务整体状态
- 阶段状态：任务时间线中每个步骤的状态

V1 服务行状态集合：

- 无任务
- 待执行
- 进行中
- 失败待处理
- 待发布
- 已完成
- 阻塞

状态映射原则：

- 同一 `服务 + 环境` 不存在两个当前状态同时竞争
- 如果有新的有效任务出现，它覆盖旧任务成为当前状态来源
- 旧任务全部归入历史任务

服务投影规则：

- 对每个 `医院 + 业务应用 + 环境 + 服务` 生成一条 `服务投影`
- 服务行展示的不是任务整体状态，而是该服务自己的 `服务投影`
- 多服务任务会为其服务范围内的每个服务分别生成服务投影
- 多个服务投影可以共享同一个 `currentTaskId`
- 但每个服务投影拥有独立的：
  - `currentPhase`
  - `status`
  - `errorSummary`
  - `canContinueBuild`
  - `canContinuePublish`

多服务任务映射规则：

- 多服务任务不会整体覆盖所有服务行的同一状态
- 任务中尚未开始的服务，其服务投影状态为 `进行中`，阶段标记为 `排队/等待执行`
- 从服务行打开任务详情时，若对应的是多服务任务，打开同一个任务抽屉，但默认聚焦当前服务的子时间线
- 服务行主动作始终只对当前服务的服务投影生效，不在服务行直接推进整条多服务任务的全部服务

多服务任务执行语义：

- V1 多服务任务只支持顺序执行，不做并发执行
- 顺序默认按用户在任务创建器中的服务排列顺序执行
- `创建并执行` 会自动尝试推进该任务的全部服务
- 任一服务进入 `失败` 或 `阻塞` 后，整条多服务任务暂停在当前服务，不继续后续服务
- 已成功完成的前序服务保持成功结果，不回滚
- 任务总体状态按“最坏优先”计算：
  - 若当前服务失败或阻塞，则任务为 `失败待处理`
  - 若仍有服务在运行或排队，则任务为 `进行中`
  - 若构建已完成但后续发布待做，则任务为 `待发布`
  - 所有服务都完成后，任务为 `已完成`

重试作用域：

- 从服务行触发 `重试当前步骤/重试整条链路`，只作用于当前服务的服务投影
- 从任务详情抽屉触发 `重试整任务` 时，才作用于整条多服务任务
- V1 不允许在服务行一键重跑整条多服务任务

当前状态来源选择：

- 对某个 `服务 + 环境`，最新的有效任务来源成为当前服务投影来源
- “最新”按 `updatedAt` 为准，缺失则退化为创建时间
- 被替换的旧任务立即降级为历史任务，不再参与服务行状态竞争

平台与本地缓存的边界：

- 平台真实状态始终高于本地缓存
- 只要平台侧已经存在可识别的任务 ID，本地状态就只能作为补充元数据，不得覆盖平台真实状态
- 本地缓存只用于：
  - 尚未拿到平台任务 ID 的瞬时过渡态
  - 应用重启后的恢复对账线索
  - 短时 UI 乐观更新

候选任务范围：

- 当前服务投影只从“当前账号在平台上可见、且可归一化到当前上下文”的任务中选择
- 这既包括 Workbench 创建的任务，也包括用户此前在平台上手动创建、但当前账号仍可见的任务
- 无法归一化到当前 `医院 + 业务应用 + 环境 + 服务` 的任务，不参与当前服务投影计算

阻塞与失败归类：

- `阻塞` 单独作为一种服务行状态存在
- 但在筛选和侧栏分组上，`阻塞` 归入 `失败待处理`

任务可见性规则：

- `待执行`
  - 表示任务已创建，但尚未启动执行
  - 可以成为当前服务投影来源
  - 在侧栏中归入 `最近任务`
- `进行中 / 等待平台结果 / 待发布`
  - 进入当前服务投影
  - 进入对应侧栏分组
- `失败待处理 / 阻塞`
  - 进入当前服务投影
  - 归入侧栏 `失败待处理`
- `已完成`
  - 仅在没有更新的有效任务时才作为当前服务投影来源
  - 同时进入历史任务

每个服务行状态至少需要以下数据：

- 当前任务 ID
- 当前阶段
- 最近执行时间
- 最近错误摘要
- 是否可继续构建
- 是否可继续发布

冲突取舍原则：

- 当“服务状态摘要数据”和“任务时间线数据”发生冲突时：
  - 若当前服务存在已识别任务 ID，则以任务时间线数据为准
  - 若当前服务不存在任务 ID，则以服务状态摘要数据为准
- 这样可以保证服务行与任务详情优先围绕同一条可追踪任务保持一致

## 8. 任务详情抽屉

所有次动作统一落到右侧任务详情抽屉。

抽屉分四个区域：

- 任务概览
- 执行时间线
- 日志与失败原因
- 历史任务

### 8.1 任务概览

展示：

- 医院
- 业务应用
- 环境
- 服务范围
- 当前状态
- 当前阶段

### 8.2 执行时间线

展示阶段节点与每段结果：

- 成功
- 运行中
- 失败
- 阻塞

### 8.3 日志与失败原因

要求：

- 直接展示最近一次失败阶段
- 展示简要错误摘要
- 支持查看完整日志

### 8.4 历史任务

展示最近若干条历史任务，并支持：

- 打开
- 克隆为新任务

V1 中历史任务至少需要保留：

- 医院
- 业务应用
- 环境
- 服务范围
- 最近执行时间
- 最终状态
- 最近失败原因

历史任务来源边界：

- 历史任务包含：
  - Workbench 创建的任务
  - 当前账号在平台上可见、且可归一化到当前上下文的任务
- 无法归一化的远端任务不进入当前工作台历史任务视图

## 9. 克隆历史任务

`克隆历史任务` 不是复制记录，而是以历史任务为模板创建新任务。

复制内容：

- 医院
- 业务应用
- 环境
- 服务范围

新任务生成后，用户可以直接再次构建和再次发布。

V1 约束：

- 单服务历史任务克隆后，默认生成新的单服务任务
- 多服务历史任务克隆后，生成新的多服务任务
- V1 不要求在克隆时修改复杂平台元信息
- 如果后端确实要求补充说明、版本号或其他字段，则由实现阶段在任务编辑器中最小化补齐

适用场景：

- 补丁测试
- 补丁发布
- 当前任务服务范围不合适，需要快速另开一单

## 10. 连接层设计

连接层不是业务模块，而是后台基础设施。

包含三类状态：

- VPN
- 构建平台
- 发布平台

### 10.1 展示方式

位于工作台顶部右侧，默认只显示简短状态。

hover 后展示：

- 当前账号
- 最近心跳
- 失效原因

状态定义：

- VPN：已连接 / 连接中 / 已断开 / 异常
- 构建平台：已登录 / 登录中 / 已失效 / 异常
- 发布平台：已登录 / 登录中 / 已失效 / 异常

判定来源：

- VPN：EasyConnect 进程状态 + 内网目标探活
- 构建平台：本地 token + 接口探活
- 发布平台：本地 token + 接口探活

### 10.2 恢复方式

异常时给统一动作：

- 一键恢复

恢复顺序：

1. EasyConnect
2. 构建平台鉴权
3. 发布平台鉴权

EasyConnect 自动化边界：

- V1 目标是全自动重连/登录，不只是拉起客户端
- 前提是初始化阶段已录入可用凭据与客户端路径
- 若 EasyConnect 当前界面可被识别且无需额外不可预期的人机校验，则执行自动登录
- 若遇到 V1 无法稳定处理的异常界面，例如未知弹窗、验证码、额外安全校验，则降级为：
  - 拉起并聚焦 EasyConnect
  - 显示明确错误原因
  - 提示用户手动完成后由系统继续后续恢复

一键恢复定义：

- `一键恢复` 默认尝试完整恢复 `VPN -> 构建平台 -> 发布平台`
- 如果 EasyConnect 恢复失败，则后续平台鉴权不继续执行
- 如果 VPN 恢复成功但平台鉴权失败，则保留 VPN 连通并分别提示构建或发布平台错误

探活与轮询策略：

- 空闲态：约 5 分钟一次轻探活，用于 token 持活
- 运行态：仅围绕当前执行中的任务提高检查频率
- 异常态：触发 fallback 流程，自动执行重试、重连、重新鉴权

默认值：

- 空闲态探活：默认 5 分钟，可配置范围 1-10 分钟
- 运行态任务检查：默认 5 秒
- 运行态异常退避上限：30 秒

设计原则：

- 平时不做高频轮询
- 只在需要跟踪当前自动链路时临时加快检查频率
- 业务异常优先靠运行态 fallback 兜底，而不是靠常驻高频探测

设计原则：

- 用户不需要把连接层当作主要操作对象
- 连接层只在异常时打断

## 11. 自动化原则

### 11.1 默认自动推进

用户点击主动作后，系统默认自动推进“当前主动作对应的目标链路”，不额外插入人工确认。

这里的“目标链路”不是固定指整条环境全流程，而是以当前主动作的目标阶段为终点：

- `dev / 构建`：目标终点是 `开发环境发布完成`
- `test / 开始构建` 或 `继续构建`：目标终点是 `待发布`
- `test / 现场发布`：目标终点是 `已完成 | 失败 | 阻塞`
- `prod / 开始构建` 或 `继续构建`：目标终点是 `待发布`
- `prod / 生产发布`：目标终点是 `已完成 | 失败 | 阻塞`
- `prod / 公司发布`：目标终点是 `公司发布完成后阻塞`

执行编排规则：

- V1 由客户端本地编排自动链路
- 编排器运行在 Electron 主进程，渲染层只负责展示和发起命令
- 不依赖额外后端编排器
- 同一 `服务 + 环境` 同时只允许一条自动链路运行
- 用户重复点击主按钮时，不新开第二条链路，而是聚焦当前任务
- 若平台接口判定当前操作不合法，则以接口错误为准，直接回填到当前任务状态

生命周期约束：

- 同一台机器同一时间只允许一个执行中的本地编排器
- 第二个应用实例只能进入观察模式，不允许发起会改变任务状态的执行调用
- 对每个 `医院 + 业务应用 + 环境 + 服务`，最多允许一个活动执行
- 活动执行在每次阶段切换前后都必须落盘快照
- 快照至少包含：
  - `taskId`
  - `serviceScope`
  - `perServicePhase`
  - `perServiceStatus`
  - 外部平台标识符
  - `lastHeartbeatAt`

应用退出与恢复：

- 正常退出时，不再启动新阶段
- 若平台侧任务已启动，本地仅停止后续推进，并将任务保留为“等待平台结果”
- 应用崩溃或被杀后，下次启动先进入恢复对账流程
- 恢复时必须先恢复连接层，再通过平台任务 ID 对账
- 禁止在未完成对账前盲目重放上一步

最小状态机：

- 编排器生命周期：
  - `Starting -> Recovering -> Running -> Stopping -> Stopped`
- 活动执行生命周期：
  - `Idle -> Queued -> Running -> Waiting -> Failed | Blocked | PendingPublish | Succeeded`

去重与接管：

- 重复点击主动作时，如果存在活动执行，则直接聚焦该任务
- 若检测到上一实例心跳超时，则新实例可接管，但接管第一动作必须是“对账”

默认接管阈值与判定：

- 心跳超时默认阈值：15 秒
- 接管判定依据：
  - 本地执行快照心跳超时
  - 原执行实例进程不可用或已失联
- 用户可见提示：
  - 接管前提示“检测到上一执行实例失联，正在接管并对账”
  - 接管失败则转为阻塞并提示手动接管

同服务多任务约束：

- 若同一 `服务 + 环境` 已存在 `Queued / Running / Waiting` 的活动任务，则禁止再创建新的执行任务
- 此时用户触发 `新建任务` 或 `克隆历史任务` 时，只允许：
  - 取消创建
  - 或创建“仅创建任务”但不执行
- 活动任务进入终态前，不允许新的执行任务替换它成为当前执行来源
- 若当前任务处于 `待执行 / 待发布 / 失败待处理 / 阻塞`，则允许用户新建任务
- 新建成功后，新的任务成为当前服务投影来源，旧任务进入历史任务

### 11.2 失败后的动作

失败后原地给出：

- 重试当前步骤
- 重试整条链路
- 手动接管

幂等语义：

- `重试当前步骤` 默认复用当前任务 ID
- `重试整条链路` 也默认基于当前任务重跑，不自动新建任务
- 只有用户主动点击 `新建任务` 或 `克隆历史任务` 时，才生成新任务

停止语义：

- V1 不提供通用的“取消整条链路”
- 用户停止自动推进时，只停止后续阶段调度
- 已提交到平台侧的构建/发布不会被隐式取消
- 若上游平台已提供“停止构建”等明确能力，V1 仅做透传

用户入口：

- `停止自动推进` 入口位于任务详情抽屉内
- 若任务在侧栏中处于进行中，侧栏可提供快速入口跳到抽屉中的停止动作

### 11.3 手动接管

仅作为兜底能力。

当平台存在异常或规则未覆盖时，系统可把用户带到对应原平台页面，但必须定位到尽可能精确的上下文，而不是让用户重新查找。

## 12. 设置页范围

V1 设置页主要负责：

- EasyConnect 客户端路径
- 网关配置
- 账号配置
- 平台地址配置
- 自动恢复与轮询策略

V1 不在设置页承载日常业务操作。

### 12.1 初始化配置

V1 第一次启动时进入初始化配置流程，录入：

- EasyConnect 相关配置
- 构建平台地址与固定账号
- 发布平台地址与固定账号
- 轮询与自动恢复策略
- VPN 探活目标与网关候选列表
- 生产自动发布 allowlist

约束：

- V1 只支持一套固定账号
- 不支持多账号切换
- 不把连接配置作为日常高频操作暴露到主工作流
- V1 仅支持 macOS

macOS 范围说明：

- EasyConnect 拉起、进程检测、路径配置均按 macOS 处理
- 不在 V1 内引入跨平台抽象

网关与探活配置：

- `vpnGatewayCandidates[]`
  - 用户可录入多个 EasyConnect 网关地址
  - V1 按顺序尝试连接
- `vpnProbeTargets[]`
  - 默认从构建平台地址和发布平台地址自动提取 host
  - 允许用户额外补充一个或多个内网探活目标
- 若用户未手动补充探活目标，则以：
  - 构建平台 host
  - 发布平台 host
  作为默认探活目标

环境映射规则：

- Workbench 内部只使用三种环境枚举：
  - `dev`
  - `test`
  - `prod`
- 各平台适配器负责把平台原始环境名或环境代码映射到这三个枚举
- 若平台环境无法唯一映射到 `dev/test/prod`，则该上下文进入阻塞，不允许自动链路继续

### 12.2 凭据处理

V1 账号固定，由初始化流程录入并存储在应用本地配置中。

约束：

- 不支持多账号
- 不支持运行时账号切换
- Keychain 集成不是 V1 前置条件

登录方式与 token 获取策略：

- 构建平台：
  - 优先采用用户名/密码直连登录接口获取 token
  - 当前已知登录链路基于第一方登录 API
  - token 失效后由连接层自动重新登录
- 发布平台：
  - 采用用户名/密码登录平台登录页
  - 从登录成功后的跳转结果中提取平台 token
  - 后续优先复用该 token 做探活与操作
- 如果任一平台在实现时发现需要 cookie 或额外会话字段，视为平台适配器内部细节，不改变产品主流程

凭据存储边界：

- V1 使用应用本地配置存储固定账号与必要凭据
- 不支持手动粘贴 token 作为主方案
- 不支持 SSO、多租户切换、多账号并存

最低安全要求：

- 敏感凭据不得以明文落盘
- Electron 必须优先使用 `safeStorage` 或等效系统支持的本地加密能力对凭据加密后再写入配置
- 敏感字段在日志中必须脱敏
- 凭据文件必须存放在应用专用数据目录，不与普通业务配置混放
- 提供明确的“重置初始化配置”能力，用于清空本地凭据与 token
- 若因版本升级导致凭据解密失败，系统必须要求用户重新录入，而不是回退到明文策略

“Keychain 集成不是 V1 前置条件”的具体含义：

- V1 不要求单独设计一套 Keychain 管理界面或自定义钥匙串条目管理逻辑
- 但允许并推荐使用 Electron `safeStorage`
- 若 `safeStorage` 在 macOS 上底层依赖系统钥匙串，这仍视为符合 V1 范围
- 若运行环境不支持 `safeStorage` 或等效系统加密能力，则 V1 必须失败关闭：
  - 不持久化凭据
  - 要求用户每次启动重新录入
  - 不允许回退到明文落盘

### 12.3 数据来源优先级

V1 规划时默认采用“接口优先，页面自动化兜底”的集成策略。

最小必需数据来源：

- 构建平台：
  - 登录鉴权接口
  - 医院/应用/服务列表接口
  - 构建任务状态与日志接口
  - 构建与公司发布动作接口
- 发布平台：
  - 登录成功后的 token 来源
  - 医院/环境/业务组发布控制台数据来源
  - 现场发布动作来源

实现优先级：

- 能直接走稳定接口的能力，优先走接口
- 无稳定接口但有可预测页面工作流的能力，允许使用受控 WebView/页面自动化兜底

### 12.4 数据刷新与对账策略

V1 的刷新策略分三类：

- 启动或上下文切换：
  - 拉取当前 `医院 + 业务应用 + 环境` 的全量服务列表
  - 拉取当前上下文下的任务摘要
  - 生成服务投影
- 空闲态：
  - 低频刷新当前上下文的任务摘要与服务投影
  - 不反复拉取全量历史记录
- 运行态：
  - 高频轮询当前活动任务
  - 仅增量更新受影响服务的服务投影

恢复对账触发条件：

- 应用启动且本地存在活动执行快照
- 执行器实例接管
- 当前活动任务在运行态发生连接异常并恢复后

## 13. V1 实施边界

### 必做

- 连接层状态与一键恢复
- 级联上下文切换
- 服务列表工作台
- 三种环境差异化链路
- 自动任务推进
- 任务详情抽屉
- 历史任务克隆

### 可后置

- 手动星标
- 复杂批量编排
- 更细致的多维筛选
- 原平台所有后台管理能力镜像

## 14. 任务创建与编辑器

V1 的任务创建以“快速执行”为主，不把编辑器做成必经步骤。

默认行为：

- 单服务无任务时的主路径：默认 `创建并执行`
- `克隆历史任务`：默认克隆为新任务，并可继续选择是否立即执行

但用户可以显式选择是否先进入编辑器。

V1 规则：

- 默认提供 `直接执行`
- 同时提供 `编辑后执行`
- 只有当接口返回缺少必要字段时，系统强制弹出最小编辑弹窗

最小编辑弹窗的存在范围：

- 不是主流程
- 只承担补字段与少量任务参数修正

V1 不预设复杂任务编辑器，但保留进入编辑态的显式入口。

### 14.1 最小编辑弹窗定义

触发条件：

- 用户主动选择 `编辑后执行`
- 平台接口返回“缺少必要字段 / 字段校验失败且可通过补字段修复”

边界：

- 单个弹窗内完成
- 不引入多步向导
- 不承载平台高级配置
- 不提供批量编排能力

字段范围：

- 永远只读：
  - 医院
  - 业务应用
  - 环境
  - 服务范围
- 可编辑：
  - 与当前动作真实生效的最小字段集合

超出范围时的降级规则：

- 若平台要求按服务分别填写字段，视为超出 V1 范围
- 引导用户拆分为多个单服务任务
- 或进入手动接管

### 14.2 多服务任务创建器

V1 的多服务任务通过统一的“任务创建器”进入。

入口：

- 服务列表工具栏的 `新建任务`
- 服务行的 `新建任务`
- 历史任务的 `克隆为新任务`

交互规则：

- 从工具栏进入：默认不预选服务
- 从服务行进入：默认预选当前服务
- 从历史任务进入：默认预选历史任务的服务范围
- 创建器始终绑定当前 `医院 + 业务应用 + 环境`
- 不允许在创建器内切换医院、业务应用或环境

服务范围选择：

- 提供服务搜索
- 提供多选复选框
- 提供“全选当前筛选结果”
- 不提供跨页复杂批量规则

提交动作：

- `创建并执行`
- `仅创建任务`

语义规则：

- `创建并执行`：创建成功后立即进入自动链路
- `仅创建任务`：只创建任务，不自动推进，由用户后续再从任务或服务行继续
- `仅创建任务` 创建出的任务状态为 `待执行`

多服务任务最小策略：

- 创建器生成的多服务任务默认按顺序执行全部服务
- 不支持在创建器里为不同服务设置不同执行策略
- 若用户只想重跑其中一个失败服务，应回到该服务行或任务抽屉中对单服务执行重试

### 14.3 必填字段策略

V1 不人为发明额外复杂业务字段，必填集合来自两部分：

- 本地通用字段
- 平台适配器返回的必填字段

计算规则：

`requiredFields = commonFields(action, env) ∪ adapterRequiredFields(action, env, context)`

本地通用字段规则：

- `直接执行`：无额外本地必填
- `编辑后执行`：允许填写任务名称和备注，但默认不强制

平台适配器规则：

- 优先在执行前做字段预检
- 若平台支持预检，则弹窗打开前拿到必填字段列表
- 若平台不支持预检，则以接口返回的可解析缺字段列表为准

最小编辑器支持的字段类型：

- 单行文本
- 多行备注
- 下拉选择
- 布尔开关

超出以上输入类型或需要逐服务分别填写时，转为手动接管

## 15. 风险与注意事项

- 构建平台与发布平台接口规则可能存在医院级差异
- 生产环境 `公司发布 -> 现场发布` 的串行前置条件必须验证稳定性
- 发布平台中可能存在“旧版本未发布”的阻塞条件，需要明确状态机处理
- EasyConnect 登录与重连需要确保不会误踢当前会话
- 原平台部分页面存在异步刷新与状态不稳定问题，集成时优先走 API，不依赖前端页面跳转

## 16. 手动接管兜底标准

手动接管的最低要求不是“打开原平台首页”，而是必须把用户带回足够精确的上下文。

最低可接受定位标准：

- 医院
- 业务应用
- 环境

理想定位标准：

- 具体服务
- 或具体任务

如果只能回到原平台首页，视为未达到 V1 的手动接管要求。

### 16.1 目标页面类型

构建平台目标页面类型：

- `BUILD_TASK_DETAIL`
- `SERVICE_BUILD_ENTRY`
- `COMPANY_PUBLISH_ENTRY`
- `BUILD_TASK_LIST`

发布平台目标页面类型：

- `RELEASE_TASK_DETAIL`
- `ONSITE_PUBLISH_ENTRY`
- `RELEASE_TASK_LIST`

选择原则：

- 总是优先跳到与当前阶段最接近的页面类型
- 缺少必要参数时，按优先级降级
- 不允许在已知上下文下直接退化到首页

### 16.2 规范化定位参数

所有手动接管优先使用一套规范化参数，再由平台适配器转换成真实 URL 或脚本化页面定位动作。

全局参数：

- `hospitalId` 或 `hospitalName`
- `appId` 或 `appName`
- `env`

服务级参数：

- `serviceId` 或 `serviceName`

任务级参数：

- `buildTaskId`
- `releaseTaskId`
- `applyId`

产物级参数：

- `artifactVersion`

### 16.3 构建平台映射标准

构建平台至少支持以下精度：

- `BUILD_TASK_DETAIL`
  - 最低参数：`buildTaskId` 或平台对应任务 ID
- `SERVICE_BUILD_ENTRY`
  - 最低参数：`hospital + app + env + service`
- `COMPANY_PUBLISH_ENTRY`
  - 最低参数：`buildTaskId` 或 `artifactVersion`
- `BUILD_TASK_LIST`
  - 最低参数：`hospital + app + env`

若平台存在真实可深链 URL，则优先使用 URL。
若平台只支持列表页筛选，则以“带筛选上下文进入并自动定位目标项”为准。

### 16.4 发布平台映射标准

发布平台在调研中表现为“单根页 + 环境卡片 + 业务组卡片/弹窗”模型，因此手动接管允许使用脚本化页面定位，不强依赖 URL 深链。

发布平台至少支持以下精度：

- `RELEASE_TASK_DETAIL`
  - 若存在平台侧发布任务 ID，则优先打开该任务详情
- `ONSITE_PUBLISH_ENTRY`
  - 最低保证：医院已选中、目标环境卡片已定位、目标业务组已展开到可执行入口
- `RELEASE_TASK_LIST`
  - 最低保证：医院和环境筛选已生效

业务组选择规则：

- 默认使用当前 `业务应用` 作为发布平台业务组来源
- 自动选择顺序为：
  - 精确匹配
  - 规范化别名匹配
  - 无法唯一确定则进入阻塞并转手动接管
- 手动接管时，至少要把用户带到：
  - 已选医院
  - 已定位环境卡片
  - 可见的业务组候选区域

### 16.5 降级规则

构建平台降级链：

- `BUILD_TASK_DETAIL -> SERVICE_BUILD_ENTRY -> BUILD_TASK_LIST`

发布平台降级链：

- `RELEASE_TASK_DETAIL -> ONSITE_PUBLISH_ENTRY -> RELEASE_TASK_LIST`

如果连 `hospital + app + env` 这组最低全局参数都缺失：

- 允许打开平台首页
- 但必须同时明文展示当前已知上下文
- 并解释缺失了哪些关键参数，导致无法精确定位
